# BTCv32 Migration Guide One-page checklist for migrating from BTCv31 → BTCv32. Includes a **Crypto Agility** section and the phased hybrid rollout plan (Imp 15). --- ## Step 0 — Read the Crypto Agility overview BTCv32 introduces a `CryptoSuiteRegistry` so the chain can evolve its cryptographic primitives without hard forks. Each suite defines a KEM, signature algorithm, and hash function. Suite IDs are stored in block headers, transactions, and stealth outputs. | suite_id | KEM | Signature | Hash | Notes | |----------|-----|-----------|------|-------| | 0 | secp256k1-ECDH | Dilithium3 | Blake3 | BTCv31 legacy | | 1 | ML-KEM-768 | ML-DSA-65 | Blake3 | **Recommended** | | 2 | ML-KEM-768 + EC | ML-DSA-65 + ECDSA | Blake3 | Hybrid (transition) | | 3 | ML-KEM-512 | ML-DSA-44 | Blake3 | Low-value / fast | | 4 | ML-KEM-1024 | ML-DSA-87 | Blake3 | High-security | | 5 | ML-KEM-768 | Falcon-512 | Blake3 | Compact signatures | --- ## Phased Hybrid Rollout (Improvement 15) ### Phase 1 — Announce (no enforcement) ```diff +BTCV32_STEALTH_PQ=true BTCV32_STEALTH_PQ_MANDATORY=false +BTCV32_STEALTH_HYBRID=true +BTCV32_PREFERRED_SUITE_ID=2 +BTCV32_SUPPORTED_SUITE_IDS=0,1,2 

Submit a governance SuiteActivationProposal via:

POST /governance/suite-proposal { "proposal_id": "activate-suite-1", "new_suite_id": 1, "old_suite_id": 0, "soft_activation_height": <current_height + 1000>, "hard_cutoff_height": <current_height + 12000>, "overlap_blocks": 10000 } 

All nodes and wallets should rebuild:

docker build --build-arg PQ=true --build-arg CRYPTO_SUITE=hybrid -t btcv32:hybrid . 

Phase 2 — Soft enforcement (both suites accepted)

After soft_activation_height is reached, suite_id=1 is active. Hybrid outputs (version=3, suite_id=2) are also accepted. Old EC outputs (suite_id=0) still work.

Wallet operators must run:

python scripts/wallet_upgrade.py --pq \ --scan-priv $BTCV32_STEALTH_SCAN_PRIV \ --scan-pub $BTCV32_STEALTH_SCAN_PUB \ --spend-priv $BTCV32_STEALTH_SPEND_PRIV \ --spend-pub $BTCV32_STEALTH_SPEND_PUB \ --pq-level 768 

Add the printed BTCV32_STEALTH_KYBER_PRIV to .env. Back it up separately from EC keys.

Monitor migration progress at GET /metrics/migration. Target: pq_migration_ratio >= 0.95 before Phase 3.

Phase 3 — Hard enforcement (EC rejected)

After hard_cutoff_height:

+BTCV32_STEALTH_PQ_MANDATORY=true +BTCV32_PREFERRED_SUITE_ID=1 +BTCV32_SUPPORTED_SUITE_IDS=1,2,3,4,5 

EC-only stealth outputs (suite_id=0) are rejected. Any wallet that has not migrated cannot receive stealth payments.

Step 1 — Unified Seed Wallet (optional, recommended)

BTCv32 supports deriving all keys from a single BIP-39 mnemonic (Imp 8). New wallets should use SeedWallet.generate(). Existing wallets can add PQ keys with wallet_upgrade.py without changing EC keys.
